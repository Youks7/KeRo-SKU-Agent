# 身份锁定的创意导演系统

本协议把真实 SKU 当作身份约束和证据源，而不是所有画面的唯一像素来源。核心原则：**商品身份严格，视觉表达自由。**

## 渐进式素材收集

按阶段收集，不用一张长清单阻塞用户：

1. 先接收产品图，识别是否为同一 SKU、有哪些视角和明显缺陷。
2. 再收集商品事实、包装、变体、品牌资产、规格和声明证据。
3. 事实边界稳定后，再接收参考详情页、风格图、场景图或竞品样本。
4. 已有 `SKU_CONTEXT`、`IDENTITY_CONTRACT`、`CREATIVE_CONTEXT` 或参考抽象报告时直接复用，只补会改变方向的缺口。

每一阶段都可以输出当前结论和下一项最重要输入。用户已提供后续阶段材料时直接处理，不强制按轮次重新索要。

## 产品图诊断与多视图统一

进入创意策划前，为每张产品图记录：用途、视角、清晰度、曝光、色偏、反光、透视、遮挡、Logo/文字可读性、包装和可见结构。

建立多视图证据板：

- 判断图片是否属于同一 SKU 与同一变体。
- 对齐颜色、比例、结构、Logo、标签、包装文字和关键表面特征。
- 为每个视角记录 `view_confidence`；没有证据的隐藏结构保持未知。
- 精修只能清洁、校色、纠正可证明的拍摄问题，不能把 AI 补全结果升级为事实。
- 图像不足不阻塞全部创意：仍可做 F1 场景合成；需要新机位的 F2 单元单独标记 `blocked` 或 `human-review`。

## 参考详情页语义分段

先按销售任务而非像素高度拆分参考页：Hero/识别、利益、场景、细节、证明、规格、比较、疑虑、品牌、收束。边界不清时先描述分段依据，不机械按固定屏数切割。

对每个模块只提取抽象视觉语言：

- 构图语法：中心、偏置、分屏、连续运动、留白和层级。
- 镜头语法：景别、机位、视角变化和跨屏节奏。
- 光线语法：方向、软硬、对比、阴影和反射。
- 材质世界：空间、表面、道具关系和环境色。
- 图形系统：网格、线条、色块、图标、字体密度和信息层级。
- 叙事节奏：情绪建立、产品证据、疑虑解决和行动收束。

输出 `参考抽象报告`：参考角色、权利状态、可提取特征、排除特征和最终生成用途。竞品参考默认 `analysis-only` 或 `prompt-only`；不得复制竞品商品、Logo、文案、人物身份、品牌故事、独特符号或一比一布局。

## CREATIVE_CONTEXT

在事实层之后、生产层之前建立 `CREATIVE_CONTEXT`：

- 商业目标、目标人群、主要购买障碍与希望产生的感觉。
- 可被视觉化的产品事实和至少两个产品专属记忆点。
- `creative_thesis`、`product_role`、`visual_world`、镜头/光线/材质/图形语法与叙事弧线。
- 各平台素材槽位的 `creative_freedom_by_slot` 和建议 F0–F3。
- 参考抽象报告、权利风险、陈词滥调和禁止视觉暗示。
- 候选方向、已选方向、被拒方向、批准状态和当前阶段。

禁止用“高级、质感、氛围、科技、奢华、极简”单独充当创意命题。命题必须说明要放大哪一项真实产品特征、解决哪一个购买任务，以及画面为何只能属于当前商品。

## 三个差异化方向

默认提出三个差异化创意方向；用户明确只要一个时也先说明选择依据。每张方向卡至少包含：

- 一句话创意命题、销售任务和目标情绪。
- 场景世界、商品角色、镜头与光线、材质环境、图形系统和叙事节奏。
- 引用的产品专属记忆点和身份处理方式。
- 适用槽位、不适用槽位、建议 F0–F3、创意风险和合规风险。
- 从参考中抽象的特征，以及明确排除的参考元素。

任意两个方向至少有三个维度不同：创意命题、场景世界、商品角色、镜头与光线、叙事节奏。只换背景颜色、道具、装饰材质或形容词视为失败，必须重写。

方向确认前只输出方向卡和少量构图草案，不输出正式逐素材生产 Prompt。用户说“按你推荐的”可视为批准，但必须记录推荐理由与默认假设。

## 策划—提示词—生成分离

严格分三层：

1. 策划：身份合同、参考抽象、方向卡、平台资产地图和跨屏视觉 Token。
2. 提示词：方向批准后，提取逐单元 Prompt、Negative Prompt、来源图、蒙版、身份锚点和发布闸门。
3. 生成：按单元调用工具或交接；最终文字、Logo、价格、参数、认证和 CTA 使用真实图层或 HTML。

参考长图默认不直接传给最终模型。只有权利明确、角色必要且工具需要时，才把合法风格/场景参考用于受控编辑，并保留参考抽象报告。

## 视觉质量闸门

每个生产单元和整套页面分别执行六道闸门：

1. 事实：参数、材质、性能、价格、包含物和声明均可追溯。
2. 身份：轮廓、比例、颜色、关键结构、Logo、标签、包装和变体符合 `IDENTITY_CONTRACT`。
3. 表现真实性：人物佩戴、反射、阴影、道具、尺度和使用关系不制造错误印象。
4. 创意质量：命题与产品事实有关，不是通用“高级背景”；相邻单元的镜头和销售任务不重复。
5. 平台：主图、SKU/variation、附图、详情、A+、PDP Section 和推广素材没有混用。
6. 参考与权利：没有复制参考商品、人物身份、Logo、文案、品牌故事、独特符号或一比一布局。

使用统一状态：`ready`、`human-review`、`direction-only`、`blocked`、`repair-required`。F2 默认 `human-review`；身份或 AI 瑕疵失败时标记 `repair-required`，不得用整体美观掩盖商品错误。

## 断点续作

`01-context/SKU_CONTEXT.json` 是唯一权威状态文件；它按 `SKU_CONTEXT V2` 同时保存事实、`IDENTITY_CONTRACT`、`CREATIVE_CONTEXT`、目标、方向批准、生产进度和 QA 状态。`IDENTITY_CONTRACT.json`、`CREATIVE_CONTEXT.json`、`reference-abstraction.json` 只能作为便于阅读或交接的导出视图，`project-state.json` 只能作为带 `state_revision` 的快速索引，任何一个都不得反向覆盖权威文件。

多平台项目在同一权威文件的 `platform_contexts` 中分别保存每个平台/站点的目标、方向、生产、工作流与 QA；`active_platform_context_id` 只选择当前工作项。切换平台不得覆盖、清空或重算其他平台的已确认状态。

保存并复用：

- `SKU_CONTEXT` 与 `IDENTITY_CONTRACT`。
- `CREATIVE_CONTEXT`、参考抽象报告和平台资产地图。
- 当前阶段、批准状态、已选/被拒方向、已完成单元、返修任务和人工审核项。

写回顺序：先在内存中合并并验证完整 `SKU_CONTEXT V2`，递增 `workflow_state.state_revision`，再用临时文件原子替换 `SKU_CONTEXT.json`；成功后才刷新可选导出视图和 `project-state.json`。不要先写缓存再写权威状态。文件系统不支持原子替换时，先保留上一版备份并明确报告。

恢复优先级：

1. 读取并验证 `SKU_CONTEXT.json`；缺失、损坏或版本不支持时停止并报告，不根据导出文件猜测完整状态。
2. 可选索引与权威文件的 `state_revision` 相同才可用于加速；索引落后时忽略并重建，索引超前或同版本内容冲突时标记 `human-review`。
3. 先按 `completed_stages` 找第一个未完成阶段；方向批准前即使已经预填 `resume_from` 或 `planned_units`，也必须停在方向确认门。只有进入 `production` 后才使用 `resume_from`，或按 `planned_units` 与 `completed_units` 计算第一个未完成单元。单平台使用顶层工作流，多平台使用 `active_platform_context_id` 指向的平台工作流。
4. 除非来源冲突、用户改变 SKU/平台/方向或证据失效，不重复产品分析、参考页分段或方向生成。

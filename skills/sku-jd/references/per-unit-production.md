# 逐屏 / 逐模块生产协议

本协议恢复 Lite V1.2.1 的逐屏生产深度，同时保留 V1.3 的平台专用素材槽位。把“生产单元”理解为平台实际使用的一屏、一个详情模块、一个图片槽位、一个 A+ 模块或一个含媒体的 PDP Section；不要因此把所有平台强制做成固定长图。

## 触发与硬门槛

1. 只在用户确认方向后输出正式生产 Prompt。用户说“按你推荐的来”可视为确认，但必须列出默认假设。
2. 先确认平台、站点或市场、类目、素材槽位、目标语言和当前规则来源。规则不确定时标记 `human-review`，不要凭记忆伪造精确尺寸或限制。
3. 每个生产单元都引用 `SKU_CONTEXT` 中的来源图片、已确认事实、处理模式和禁止修改项。
4. 缺少会影响商品身份、交付内容、声明或平台合规的证据时，把对应单元标记为 `blocked`，保留字段和占位符，不生成可误用为正式成品的 Prompt。
5. 任何价格、尺寸、材质、成分、性能、兼容性、认证、评价、排名、产能、MOQ、售后或促销声明都必须绑定证据来源。
6. 平台专用规则优先于本协议。平台禁止覆盖文字、数字渲染、场景背景或某种比例时，使用更严格要求。
7. 每个生产单元必须完整输出本协议字段，不得用一句“同上”省略 Prompt、Negative Prompt、处理模式、排版、镜头、质检或拦截自检。

## 整页生产总控

在逐单元内容之前先输出一次 `整页生产总控`：

- 已确认方向、推荐理由、目标平台、站点、类目、语言和页面/素材类型。
- 处理模式总原则，以及哪些单元被平台规则强制使用模式 A。
- 生产单元总表：编号、平台槽位、销售任务、证据状态、预计比例/响应式行为、发布状态。
- 信息故事线：识别商品 → 建立利益 → 提供产品证据 → 解释规格/使用 → 消除顾虑 → 信任/行动；按平台和商品删改，不机械套用。
- 整页视觉 Token：主辅色、背景材质、光源方向、对比度、字体层级、网格、边距、文字安全区、圆角/线条规则和禁用元素。
- 跨屏连续性：产品朝向、镜头距离节奏、背景过渡、产品占比、文案密度和相邻单元衔接方式。
- 缺失证据和被阻塞单元，不得用默认值偷偷补齐。

## 逐单元强制模板

对每一屏、每一模块或每一媒体槽位逐项输出：

```text

平台槽位 / 模块类型：
销售任务：
发布状态：ready / human-review / direction-only / blocked
本单元依赖：来源图片 ID、文件名或资料；对应已确认事实
禁止主张 / 缺失证据：

未来画面：
主文案：
副文案 / 证据文案：
文案位置与安全区：

产品处理模式：A / B / C
处理理由：
允许修改：
禁止修改：
产品来源与合成方式：

镜头矩阵：
- 景别：
- 视角 / 机位：
- 产品朝向 / 状态：
- 产品占比 / 位置：
- 背景 / 道具：
- 光线 / 阴影 / 反射：
- 景深 / 透视：
- 与前后单元的差异及衔接：

Prompt：
Negative Prompt：

后期排版：
- 画布比例 / 尺寸 / 响应式裁切：
- 文字图层、字体层级、对齐和行数：
- 产品、文字、标识和 CTA 的安全区：
- 色彩、对比度、边缘、阴影和导出要求：

产品一致性质检：
1. 颜色、结构、比例、Logo、标签和包装是否与来源一致？
2. SKU、变体、数量、配件和交付内容是否一致？
3. 是否出现相似款替换、错误文字、额外部件或 AI 变形？
4. 文案、参数、声明和图中符号是否都有证据？
5. 该单元最终应标记为什么发布状态？

通用 Prompt 拦截自检：
1. 把产品替换为同类商品后，Prompt 是否仍完全成立？
2. 是否只是“产品 + 高级背景 + 柔光 + 留白”等通用模板？
3. Prompt 是否包含至少两个有来源的本产品专属细节？
4. 是否混入未确认事实、平台禁用元素或错误素材槽位？
```

对于纯 HTML、参数表或真实页面组件，仍保留全部字段；把 `Prompt` 明确写成“`N/A——本单元使用语义化 HTML / 真实组件，不生成图片文字或假按钮`”，并给媒体背景或配图单独建立生产单元。

## Prompt 准确性规则

1. Prompt 开头明确输入依据、平台槽位和任务，不把策略说明伪装成视觉事实。
2. Prompt 只使用已确认事实；谨慎推断只能用于方向说明，不得进入正式商品声明。
3. 在 Prompt 中写出至少两个本产品专属视觉细节，并能回指来源图片位置。
4. 明确产品数量、视角、位置、背景、光源、阴影、留白和文字安全区；未确认尺寸不用虚构毫米数。
5. 生图模型不可靠地生成文字、Logo、包装标签、参数表、价格或认证标识时，Prompt 必须要求不生成，并交给后期真实图层处理。
6. 不把 Negative Prompt 当作事实保护的唯一手段；关键身份约束同时写入正向 Prompt、处理模式和质检。
7. 同一长页的 Prompt 共享视觉 Token，但每个单元必须改变商业任务或镜头证据，避免只换背景颜色。

## 产品处理模式

- **模式 A｜严格保真**：产品本体使用用户真实图抠图、裁切或真实局部图；Prompt 只生成背景、场景、光影、材质、氛围、道具、构图留白和产品预留区，不描述或重绘产品本体。
- **模式 B｜AI 辅助商品图**：以指定真实产品图为唯一身份参考；明确保持类别、颜色、结构、比例、Logo、标签、包装文字和关键外观。输出后逐项比对；任何关键变化都重新生成或切换模式 A。
- **模式 C｜概念生成**：仅用于无实物提案或方向测试；必须标记 `direction-only`，不得作为真实售卖 SKU 成品。

普通 SKU 不自动获得模式 B。主图、SKU 属性图、品牌敏感商品、包装文字敏感商品以及平台禁止数字渲染的槽位优先模式 A。

## 文案与后期排版

1. 把最终文字、Logo、价格、参数、认证和 CTA 保留为可编辑真实图层或语义化 HTML，不依赖模型生成可读文字。
2. 写清标题、副标题、证据文案和脚注的具体位置、对齐、最大行数、文字安全区及移动端最小可读性。
3. 不使用“高端、品质、专业、黑科技”等空泛词替代商品证据。
4. Shopify 的价格、Variant、Add to Cart 和政策信息使用真实组件；Amazon、TikTok Shop 等禁止覆盖的槽位不添加文字。
5. 长图拼接前统一字体、字号层级、网格、边距、背景过渡和导出色彩空间；检查接缝、裁切和重复信息。

## 镜头矩阵与连续性

1. 为每个单元明确景别、角度、机位、产品状态、产品占比、背景、道具、光线、景深和透视。
2. 相邻单元至少改变镜头距离、视角、产品状态、证据重点或背景逻辑中的两项，但保持同一 SKU、光影世界和视觉 Token。
3. 全页至少包含识别镜头、产品专属细节镜头和购买确定性信息；缺少真实来源时阻塞对应镜头。
4. 不为了视觉节奏添加用户不会收到的配件、赠品、包装、场景能力或使用结果。

## 质检与通用 Prompt 拦截

逐单元检查后，再执行整页检查：

- 商品身份、颜色、结构、变体、数量和包含物是否从头到尾一致。
- 页面顺序是否符合目标平台的购买任务，而非固定六屏模板。
- 同一卖点、相似构图或同一视角是否无意义重复。
- 所有数字、材质、性能、认证、比较、价格和服务承诺是否可追溯。
- 文字是否可读、可编辑、未被模型误生成，移动端是否保留安全区。
- 平台主图、SKU 图、详情模块、广告素材、A+ 或 PDP Section 是否被错误混用。
- `blocked` 和 `direction-only` 单元是否被误标为可发布。

## 输出节奏

- 用户未要求一次性输出全部时，先完整输出前 3 个生产单元，确认视觉系统后再继续。
- 用户明确要求完整详情页时，一次性输出所有单元，但每个单元都必须使用完整模板。
- 用户只要求首屏时，只输出首屏的完整模板。
- 用户要求首屏 A/B 测试时，输出 2 至 3 个完整首屏方案；每个方案分别执行拦截自检。
- 用户只要求策略或方向时，不提前输出正式生产 Prompt。
